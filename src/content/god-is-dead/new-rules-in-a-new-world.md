Being a software engineer who works in startups placed me in a privilledged position of standing at the frontier of all 
the changes that the storm of AI brings with itself. If there are still areas where this monster of a technology has not
penetrated yet, software engineering is transforming literally by day. And its future form is still in the making.

Half year ago, I was still just consulting ChatGPT when I could not find instantly a proper solution for my code. Since
November 2025, I have not written a single line of code myself. I am giving tasks to several agents that run in parallel
and plan other tasks while the agents are working. I still feel nostalgic for the old good days where you could sit for 
hours on your code, solvign a problem, falling into a meditative trans where nothing else exists but you and code... 
These days are gone forever, however.

So the first change that is in the making -- is the definition of software engineering job. Before it was clear --
main task of a software engineer was to produce code. Today, you do not really have to be a seasoned software engineer
to write a pretty good code. Even people who do not know any programming language can write code, and thanks to LLMs,
its quality can be not necessary much worse than the code of seasoned engineers who wrote this code half year ago by hand.
Thus, producing code is not anymore a privillidged skill that can solely form a profession.

So what does the job of a software engineer entail today? Is it about designing digital systems? Ability to deploy them 
and scale? Or it comes to the role of a technical produc owner? This question remains open. In any case, whatever the 
definition of software engineer is 2026, it is not the same as it was just half year ago.

Second change is software development processes. Before, the same workflow existed for decades: spec is written, 
design is made, code implementation of the design and spec, code review, qa, deployment. You could follow a book. Today, 
the book went to bin. Ai made writing code so fast and cheap, that specs, design, reviews, qa become bottlenecks. There
is no widely accepted recipe yet how to address it. In my past company, we decided to swap design with coding, so that 
design can de developed later not to block features development. It made sense. Then, however, we decided to fire designer
as "AI became good enough to do the job...". The design became much shittier tbh, but the management was happy to
consider it as efficiency improvement. 

PR reviews became a joke. I felt sometimes that my AI was talking to the AI of the code reviewer, making me wonder why 
not just to connect our agents and let them do it without the barrier of human intervention. Same holds for most of 
technical discussions: people are throwing their prompt results to each other, measuring whose prompt is better.
